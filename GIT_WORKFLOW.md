# Git 协作说明

## 推荐分支策略

- `main`：始终保持可运行、可展示
- `feature/member-a-data`
- `feature/member-b-extraction`
- `feature/member-c-graph`
- `feature/member-d-ui`

如果你们不想开太多分支，至少保证：

- 日常开发不要直接往 `main` 里乱推
- 每次合并前先确认本地可运行

## 提交前必须检查

1. 页面是否还能打开
2. `app.py` 是否还能正常启动
3. 是否误提交了缓存文件、私钥、`.env`
4. 是否误提交了临时生成的大体积 JSON/TTL

## 推荐提交粒度

- 一次提交只做一类事情
- 不要把前端样式、后端逻辑、实验草稿混在一个提交里

例如：

- `feat: 新增 SPARQL 查询工具骨架`
- `feat: 新增普通问答页面布局`
- `docs: 更新成员 D 整合说明`
- `fix: 修复空 SPARQL 请求报错`

## 推荐协作方式

- 成员 B 改抽取逻辑时，不直接改成员 D 的前端展示
- 成员 C 改 schema 时，必须同步通知成员 D 更新查询模板
- 成员 D 如果改了接口字段，要同步写进文档

## 不建议提交到仓库的内容

- 本地调试缓存
- `.env`
- 私有 API Key
- 未清洗的巨大中间文件
- 临时截图或一次性测试输出
